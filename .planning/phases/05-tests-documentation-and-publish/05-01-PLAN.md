# Phase 05 Plan 01: Fill nix flake check Gaps

## Objective

Add 2 targeted `nix flake check` derivations that cover code paths not yet tested:
1. `unsupported-version-error` — verifies that requesting an unsupported runtime version throws a descriptive error
2. `integer-env-var` — verifies that bare integer TOML env values are coerced to strings by `builtins.toString`

These are the only meaningful gaps in the existing 10-check suite identified in the Phase 05 research.

## Context

- `flake.nix` — 10 existing checks; the `unknown-tool-error` pattern (tryEval + deepSeq) is the model for the new unsupported-version-error check
- `lib/runtimes.nix` — throw message: `"mise2nix: node version 18 not available in nixpkgs — supported: 20, 22, 24, 25"`
- Both new checks use inline `builtins.toFile` fixtures (no new .toml files in repo)

## Tasks

### Task 1: Add unsupported-version-error check

Add the `unsupported-version-error` check to `flake.nix`. This replicates the `unknown-tool-error` pattern using `builtins.tryEval (builtins.deepSeq devShell.nativeBuildInputs devShell)` to force lazy evaluation and catch the throw.

The TOML fixture uses `node = "18"` — node 18 is explicitly not in the runtime map (supported: 20, 22, 24, 25), so `resolveNode` will throw.

**Done when:** The check appears in `flake.nix` and `nix build .#checks.x86_64-linux.unsupported-version-error --no-link` succeeds.

<automated>
nix build .#checks.x86_64-linux.unsupported-version-error --no-link
</automated>

### Task 2: Add integer-env-var check

Add the `integer-env-var` check to `flake.nix`. This replicates the `env-var-passthrough` pattern but uses a bare integer TOML value (`PORT = 8080` without quotes) to verify `builtins.toString` coercion in `lib/env.nix`.

**Done when:** The check appears in `flake.nix` and `nix build .#checks.x86_64-linux.integer-env-var --no-link` succeeds.

<automated>
nix build .#checks.x86_64-linux.integer-env-var --no-link
</automated>
